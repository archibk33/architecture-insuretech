# Task1

Технологическая архитектура to be для InsureTech

## Цель

Обеспечить отказоустойчивость и равномерное время отклика для пользователей в регионах РФ при росте нагрузки. Выполнить RTO 45 минут и RPO 15 минут. Достичь доступности 99.9%.

## Ключевые решения

* Мультизональная и мультирегиональная архитектура с активным приложением в 2 регионах
* Независимые кластеры Kubernetes по одному на регион без растягивания одного кластера между регионами
* Глобальная балансировка трафика по задержке и по здоровью через глобальный DNS с низким TTL и через внешние балансировщики уровня 7
* CDN для статических ресурсов фронтенда и кеширование ответов чтения на периметре
* Внутренняя балансировка через ingress контроллер и сервис мэш по необходимости
* Реляционная база данных PostgreSQL в управляемом кластере с синхронной репликацией между зонами 1 региона и асинхронной репликацией в резервный регион
* Локальные read replica в каждом регионе для чтений и снижение задержек
* Резервное копирование и восстановление до точки времени через хранение WAL и регулярные бэкапы
* Кеш уровня приложения в каждом регионе через Redis для ускорения чтений и снижения нагрузки на базу
* Ограничение запросов и квоты для партнеров через API шлюз и WAF

## Топология to be

Приложение развернуто в двух регионах. В каждом регионе 2 зоны доступности. В каждом регионе независимый кластер Kubernetes управляемого типа. Между пользователем и кластерами работает глобальный балансировщик. Для статических файлов включен CDN. Динамические запросы направляются в ближайший регион по задержке. Сервисы внутри кластера доступны через ingress. Здоровье сервисов проверяется через эндпоинт health в каждом сервисе.

## Кластеры Kubernetes

Выбор независимых кластеров по регионам. Причина выбор лучшей устойчивости к сетевым сбоям между регионами и простоты управления. Растянутый кластер не используется из за высокой задержки и рисков сплит брейн. В каждом кластере включен autoscaling. Используются HPA по CPU и памяти. Установлены запросы и лимиты ресурсов. Включены под disruption budget и topology spread для равномерного распределения по зонам.

## Балансировка и health check

На периметре глобальный DNS с проверками здоровья регионов. TTL двадцать секунд. Локально в регионе входной трафик принимает L7 балансировщик. Ingress контроллер выполняет проверку здоровья по пути health. Для партнерского API включены отдельные лимиты RPS и burst. Включен circuit breaker и таймауты. Для клиентов веб установлен WAF и защита от DDoS.

## Фейловер стратегия

При потере одной зоны внутри региона локальный балансировщик исключает нездоровые зоны. При потере целого региона глобальный балансировщик переводит весь трафик в живой регион. Цель восстановления приложений не более сорок пять минут благодаря независимым кластерам и IaC. Данные доступны через асинхронную реплику. Допустима потеря данных не более пятнадцать минут. Переключение роли базы производится автоматизированно. Приложения перенастраиваются на новый мастер через строку подключения в менеджере секретов.
Выбранная стратегия: Active-Standby (Warm Standby) с георезервированием и асинхронной репликацией.

## База данных

Переход с одиночной ВМ на управляемый кластер PostgreSQL. В основном регионе один мастер и один синхронный стендбай в другой зоне. Это защищает от отказа зоны без потери данных. Во втором регионе одна асинхронная реплика для чтений и для аварийного переключения. Включено хранение WAL в объектном хранилище и восстановление до точки времени. Политика бэкапов ежедневно и ретеншен тридцать дней. В приложении включены стратегии повторов с экпоненциальной паузой и идемпотентность для критичных операций.

## Шардирование

Текущий объем данных 50 гб. Шардирование не требуется. Для будущего роста применить секционирование таблиц по времени для историй заявок. Это упростит обслуживание и ускорит архивные запросы.

## Равномерное время загрузки

Статика фронтенда обслуживается через CDN. Для данных чтения используется локальная реплика и кеш Redis в каждом регионе. Кэшируемые ответы имеют короткий срок жизни. Динамические записи идут в основной регион. Для экранов с тяжелыми запросами применяются материализованные представления и фоновые обновления. Это снижает задержку.

## Безопасность и доступ партнеров

Партнерский API вынесен на отдельный домен и вход через API шлюз. Включены аутентификация по ключам и квоты на клиента. Лимиты не позволяют одному партнеру занять все ресурсы. Правила WAF закрывают тривиальные атаки. Логи и метрики отправляются в централизованный стек.

## Наблюдаемость и алерты

Сбор метрик Prometheus и трассировка. Логи в централизованное хранилище. Алерты по доступности и ошибкам приложений. Алерты по задержке и очередям. Настроены каналы оповещения команды дежурных. Это ускоряет реакцию на инциденты.

## Что на схеме

На схеме отражены регионы и зоны. Глобальный DNS и CDN. Балансировщики на периметре. Два кластера Kubernetes. Ingress контроллер. Сервисы core app и другие сервисы. Redis в каждом регионе. Кластер PostgreSQL с синхронной репликацией в пределах региона и асинхронной репликой во втором регионе. Потоки трафика. Точки health check. Механизмы фейловера. Подписи о причинах выбора независимых кластеров и выбранной стратегии репликации.

## Обоснование

Решение выполняет RTO и RPO благодаря управляемому кластеру базы и автоматизации. Доступность 99.9 достигается мультизонным развертыванием и глобальным фейловером. Равномерное время загрузки обеспечивается CDN и локальными чтениями из реплик и кеша. Стоимость контролируется через autoscaling и кеширование. Сложность снижена за счет независимых кластеров по регионам.

## Дальнейшие шаги

Подготовить схему в draw.io с элементами Yandex Cloud. Описать комментариями на схеме выбранные варианты для кластеров и базы. Проверить соответствие нефункциональным требованиям на перечне сценариев отказов.

## Проверка соответствия требованиям

RTO 45 минут. Обеспечивается автоматизацией разворачивания кластеров и описанием инфраструктуры как код. При отказе региона переключение трафика и восстановление сервисов укладывается в целевое время.

RPO 15 минут. Обеспечивается асинхронной репликацией между регионами и непрерывной архивацией журналов транзакций. Потеря данных ограничена окном репликации.

Доступность девяносто девять и девять десятых процента. Обеспечивается мультизонным развертыванием, автоматическим фейловером и autoscaling. Health check на всех уровнях снижает время обнаружения сбоев.

Равномерное время отклика для регионов. Обеспечивается CDN для статики и чтениями из локальных реплик и кеша в каждом регионе. Динамические записи идут в основной регион без заметной деградации за счет очередей и идемпотентности.
